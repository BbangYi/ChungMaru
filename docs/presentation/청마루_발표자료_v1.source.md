## 01. 청마루
문맥 기반 실시간 유해 텍스트 중재 시스템

## 02. 문제 정의
욕설 필터링은 단어 탐지가 아니라 문맥, 플랫폼, 실시간성 문제입니다.
- 같은 단어도 직접 공격, 사전 설명, 고유명사, 뉴스 인용, 검색 추천어에서 의미가 달라집니다.
- Chrome DOM, Android 접근성 트리, YouTube 썸네일처럼 수집 경로가 서로 다릅니다.
- 정상 문맥을 과하게 가리면 정보 접근성과 신뢰성이 떨어지고, 늦게 가리면 보호 효과가 줄어듭니다.

## 03. 기존 접근의 한계
키워드 필터와 모델 단독 판단은 실제 화면 적용에서 각각 실패합니다.

## 04. 청마루의 핵심 차별점
허용 / 경고 / 마스킹 / 보류로 나누는 evidence 기반 중재 파이프라인입니다.
- backend 모델 판정을 최종 기준으로 유지합니다.
- evidence_spans가 있는 실제 근거 구간만 화면에 적용합니다.
- 오탐, 미탐, 지연, UI 깨짐을 regression과 이력으로 관리합니다.
- Chrome Extension과 Android 접근성/OCR 경로를 함께 검증합니다.

## 05. 전체 아키텍처
수집 계층, backend 판단 계층, 화면 중재 계층을 분리합니다.

## 06. Backend 분석 계약
공개 API 계약을 흔들지 않고, evidence와 진단을 중심으로 품질을 관리합니다.
- `/analyze_batch`: Chrome extension 중심 배치 분석 경로
- `/analyze_android`: Android 접근성/OCR 후보 분석 경로
- 라벨: profanity, toxicity, hate 중심의 scores와 boolean 판정
- 화면 적용 기준: boolean + evidence_spans 유효성

## 07. Backend 모델 평가 요약
총 1007건 기준 정확도 78.55%, 실패 216건을 분석했습니다.

## 08. 실패 유형 1: 고확신 오탐
KOLD clean false positive가 전체 FP의 대부분을 차지했습니다.
- kold_clean FP 72건: 전체 FP 86건 중 84% 수준입니다.
- 차별금지법, 성소수자처럼 주제 단어를 hate 신호로 과하게 학습한 사례가 확인됐습니다.
- 문맥 후처리와 regression set이 필요한 이유입니다.
- chart source: docs/presentation/materials/backend-model-failure-summary-20260506.md

## 09. 실패 유형 2: 낮은 점수 미탐
우회 표현, 간접 공격, 비꼼은 모델 점수만으로 잡기 어렵습니다.
- offensive_other FN 55건, racial_hate FN 21건, gender_hate FN 17건이 큰 축입니다.
- 초성, romanized/qwerty, 은어, meme 표현은 정규화와 evidence 투영이 필요합니다.
- 단순 사전 추가만으로 해결하면 문맥 오탐이 다시 증가합니다.
- chart source: docs/presentation/materials/backend-model-failure-summary-20260506.md

## 10. 임계값 조정의 한계
높은 점수 오탐과 낮은 점수 미탐이 동시에 존재합니다.
- >=0.70 구간에도 FP 66건이 있어 threshold를 올리는 것만으로 안전하지 않습니다.
- 0.30~0.50 구간 FN 85건은 threshold를 낮추면 오탐을 크게 늘릴 수 있습니다.
- 따라서 모델 점수 + 정규화 + safe-context + evidence 검증이 함께 필요합니다.
- chart source: docs/presentation/materials/backend-model-failure-summary-20260506.md

## 11. 파이프라인 비교 프레임
모델 하나의 점수보다 어떤 단계가 어떤 실패를 줄였는지 봅니다.

## 12. Chrome Extension 설계
웹에서는 DOM 수집과 exact span 렌더링을 중심으로 접근했습니다.
- visible container, active input, high-signal text를 우선 분석합니다.
- foreground와 reconcile을 분리해 빠른 반응과 문맥 보정을 동시에 노립니다.
- 전체 카드/문단/URL을 덮지 않고 evidence span만 적용합니다.
- Google AI Overview와 동적 SFC 영역은 rescue pass로 보완합니다.

## 13. Chrome에서 확인한 제약과 개선
입력창, sensitivity, cache, stale response가 실제 품질을 좌우했습니다.

## 14. Android 수집 전략
Android는 DOM span이 없으므로 Accessibility-first로 시작해야 합니다.
- TextView, 댓글, 제목처럼 접근성 텍스트가 있는 경우 OCR 없이 backend로 보냅니다.
- contentDescription-only 카드와 이미지 내부 글자는 좌표 신뢰도가 낮아 별도 경로로 분리합니다.
- boolean positive와 실제 마스킹 가능한 evidence span을 분리해 진단합니다.
- YouTube/Chrome/일반 앱을 ScreenTextCandidate 형식으로 정규화합니다.

## 15. Android OCR 전략
OCR은 판정자가 아니라 text + bbox 수집기입니다.

## 16. 마스킹 UI 원칙
잘못 빠르게 가리는 것보다, 근거 있는 위치만 안정적으로 가리는 것이 우선입니다.
- 금지: 카드 전체, 썸네일 전체, paragraph 전체, visible band fallback masking
- 허용: backend evidence span을 실제 DOM range 또는 OCR bbox로 투영 가능한 영역
- 스크롤/mutation 중 stale mask는 폐기하고 stable frame에서 재분석합니다.
- 입력창 UX는 별도 fixed-token 방식으로 관리합니다.

## 17. 협업과 이력 관리
공식 이력은 GitHub, 반복 추적은 Linear, 보고서 요약은 docs/Notion으로 분리합니다.

## 18. 현재 구현 상태
완료된 것과 검증 중인 것을 분리해 발표합니다.

## 19. 향후 개선 계획
품질 향상은 모델 재학습, OCR 안정화, QA matrix, 성능 계측으로 이어집니다.
- 모델: topic bias 제거, 우회 표현/간접 공격 데이터 보강, regression set 확대
- Chrome: 동적 DOM rescue 유지, latency/cache/stale 지표 기반 회귀 검증
- Android: ROI OCR 실험, scroll 중 stale mask 제거, 접근성/OCR 후보 분리
- 보고서: 문제-제약-선택-검증 이력을 GitHub/Linear/docs로 연결

## 20. 결론
청마루는 유해 단어를 숨기는 도구가 아니라, 문맥과 플랫폼 제약을 관리하는 중재 시스템입니다.

# 발표자 노트 / 보고서 재사용 문장

## 01. 표지
- 발표 핵심 문장: 청마루는 단어 하나를 숨기는 확장 프로그램이 아니라, 웹과 모바일 화면에서 유해 텍스트를 수집하고 문맥 기반으로 판단한 뒤 화면에 안정적으로 중재하는 시스템입니다.
- 강조점: Chrome Extension, Android App, FastAPI Backend가 분리되어 있고, 단순 키워드 필터가 아닌 evidence 기반 판단과 마스킹을 목표로 합니다.

## 02. 문제 정의
- 실제 문제는 욕설 단어 자체보다 "이 단어가 지금 누구에게 어떤 맥락으로 노출되는가"입니다.
- 예시: `시발`은 직접 공격 문장에서는 차단해야 하지만, `시발 자동차`, `Kapil Sibal`, 사전 설명, 뉴스 인용에서는 무조건 차단하면 오탐입니다.
- 사용자 경험 측면에서는 너무 늦게 가리면 보호 효과가 없고, 너무 넓게 가리면 정보 접근성을 해칩니다.

## 03. 기존 접근의 한계
- 키워드 기반은 빠르지만 문맥을 보지 못해 사전/고유명사/설명문을 과도하게 가립니다.
- 모델 기반은 문맥을 볼 수 있지만 실시간 화면에 붙일 때 latency, stale result, span 좌표 문제가 발생합니다.
- 단순 후처리는 특정 사이트에는 맞을 수 있으나 Chrome, YouTube, WebView, 일반 앱까지 확장하면 유지보수가 어렵습니다.

## 04. 청마루의 차별점
- 청마루는 "차단/허용"만 두지 않고, 허용, 경고, 마스킹, 보류로 상태를 나누어 다룹니다.
- 최종 판정은 backend 모델과 evidence span을 기준으로 유지하고, frontend는 그 결과를 화면 구조에 맞게 투영합니다.
- 오탐과 미탐은 추상적인 감상이 아니라 regression case와 실패 원인으로 남겨 개선합니다.

## 05. 전체 아키텍처
- 수집 계층은 플랫폼별입니다. Chrome은 DOM/visible container, Android는 Accessibility/OCR 후보를 수집합니다.
- 판단 계층은 backend API입니다. `/analyze_batch`와 `/analyze_android`는 텍스트 후보를 받아 라벨, 점수, evidence span을 반환합니다.
- 중재 계층은 화면 렌더링입니다. DOM Range, editable input, Android overlay 등 플랫폼별로 다른 제약을 갖습니다.

## 06. Backend 분석 계약
- API 계약을 흔들지 않는 것이 중요했습니다. extension과 Android가 같은 모델 의미론을 공유해야 하기 때문입니다.
- `evidence_spans`가 없는 positive는 화면에 정확히 가릴 위치가 없으므로 즉시 마스킹 대상으로 쓰기 어렵습니다.
- 따라서 boolean 판정과 evidence 유효성을 분리해 진단하고, frontend sensitivity가 backend 판정을 임의로 뒤집지 않도록 관리합니다.

## 07. Backend 모델 평가 요약
- 분석 데이터 기준 정확도는 78.55%였고, 실패는 216건이었습니다.
- 이 수치는 모델이 완전히 실패했다는 의미가 아니라, 실제 서비스에서는 오탐/미탐 유형을 분해해서 다뤄야 함을 보여줍니다.
- 발표에서는 정확도 하나보다 FP/FN의 원인을 나누어 설명하는 것이 설득력이 높습니다.

## 08. 고확신 오탐
- 가장 큰 오탐 축은 KOLD clean 문장입니다.
- 정치/사회/정체성 주제어가 hate 신호로 잘못 학습된 경우가 있어, threshold만 조절하면 해결되지 않습니다.
- 해결 방향은 safe-context regression set, topic bias 완화, span sanity filter입니다.

## 09. 낮은 점수 미탐
- 미탐은 직접 욕설보다 우회 표현, 간접 공격, 은어, 초성/로마자 변형에서 많이 발생합니다.
- `Tlqkf`, `ssibal`처럼 입력 방식이 바뀐 표현은 normalized variant와 raw token 위치 매핑이 필요합니다.
- 단순 사전 확장은 빠르지만 오탐을 증가시키므로 모델/정규화/evidence projection을 같이 설계해야 합니다.

## 10. 임계값 조정의 한계
- 높은 점수 구간에도 오탐이 있고, 낮은 점수 구간에도 미탐이 존재합니다.
- 민감도 슬라이더는 사용자 설정 UI로 필요하지만, 모델 품질 문제를 slider 하나로 해결할 수는 없습니다.
- 따라서 threshold는 보조 수단이고, 문맥 후처리와 regression 관리가 핵심입니다.

## 11. 파이프라인 비교
- 교수님 피드백에 대응하려면 "AI를 붙였다"가 아니라 어떤 pipeline이 어떤 실패를 줄였는지 비교해야 합니다.
- 비교 축: keyword-only, ML-only, ML+normalization, ML+safe-context, ML+safe-context+evidence projection, full pipeline.
- 각 단계별로 accuracy, FP, FN, latency, span 적용 성공률을 기록하면 보고서와 발표 모두에 사용할 수 있습니다.

## 12. Chrome Extension 설계
- Chrome은 DOM이 있으므로 OCR보다 DOM 기반 exact span이 우선입니다.
- Google 검색 결과는 동적 DOM, AI Overview, SFC 영역처럼 일반 selector로 놓치는 영역이 있어 rescue pass가 필요했습니다.
- 전체 카드 마스킹은 금지하고, evidence가 실제 텍스트 노드와 겹칠 때만 적용하는 방향으로 정리했습니다.

## 13. Chrome 제약과 개선
- 가장 큰 제약은 입력창, 캐시, stale response, sensitivity 변경 시 race입니다.
- 입력창은 일반 DOM Range로 처리할 수 없어 별도 fixed-token UX가 필요했습니다.
- 새 페이지 진입, 새로고침, 스크롤, mutation에서 동일 후보가 반복 분석되며 flicker가 생겨 generation/fingerprint 검증이 필요했습니다.

## 14. Android 수집 전략
- Android는 Chrome처럼 DOM span이 없으므로 AccessibilityNodeInfo에서 텍스트와 bounds를 얻는 것이 1차 경로입니다.
- 그러나 YouTube 썸네일/영상 프레임의 글자는 이미지 픽셀이라 접근성 트리에 없을 수 있습니다.
- 그래서 `ScreenTextCandidate` 형태로 접근성 후보와 OCR 후보를 같은 데이터 모델로 묶는 방향이 필요합니다.

## 15. Android OCR 전략
- OCR은 유해성 판정자가 아니라 `text + bbox` 수집기입니다.
- 모든 화면에 OCR을 돌리면 배터리와 latency가 무너집니다. 따라서 접근성 텍스트를 먼저 쓰고, 이미지/썸네일/영상 프레임만 ROI 제한 OCR로 처리합니다.
- scroll 중에는 OCR 마스크를 이동시키지 않고 폐기한 뒤 stable frame에서 다시 붙이는 것이 stale mask 방지에 더 안전합니다.

## 16. 마스킹 UI 원칙
- 잘못된 위치에 빠르게 가리는 것보다, 근거 있는 위치만 안정적으로 가리는 것이 더 중요합니다.
- 금지해야 할 fallback: 카드 전체, 썸네일 전체, paragraph 전체, contentDescription 전체 bounds.
- 허용 가능한 마스크: DOM Range, OCR word/line bbox, 짧고 신뢰 가능한 title/input token.

## 17. 협업과 이력 관리
- 공식 이력은 GitHub PR과 commit으로 남기고, Linear는 문제/실험/검증 루프를 연결하는 보조 이슈로 씁니다.
- 문서는 많게 만들지 않고 `constraints-and-decisions.md`, `engineering-history.md`, `reporting-workflow.md`를 중심으로 유지합니다.
- 보고서에는 "문제 - 제약 - 선택 - 검증 - 남은 리스크" 순서로 정리하면 단순 구현물이 아니라 엔지니어링 개선 과정으로 보입니다.

## 18. 현재 구현 상태
- Backend: 기본 API 계약, 실패 분석, normalization/pattern/regression 보강이 진행되었습니다.
- Chrome Extension: DOM exact span, 입력창 UX, Google 동적 영역 대응, stale/sensitivity 진단을 반복 개선했습니다.
- Android: 접근성 기반 후보 추출, overlay 진단, visual OCR 경로의 초기 구조를 만들었으나, 실시간 visual masking 품질은 아직 연구/개선 대상입니다.

## 19. 향후 개선 계획
- 모델: topic bias 완화, 우회 표현/간접 공격 데이터 보강, KOLD clean FP와 romanized FN 회귀셋 확대.
- Extension: 새 페이지 진입 직후 first-mask latency, AI Overview/SFC rescue coverage, sensitivity 적용 일관성 개선.
- Android: Accessibility-first + OCR-for-geometry + visual OCR fallback 구조로 재정렬하고, stale mask와 ROI sanity check를 강화.
- 운영: PR 단위로 문제/해결/검증을 남기고, 발표/보고서에는 주요 의사결정만 요약합니다.

## 20. 결론
